根因穿透 · 三层分析
L1
表层
App 后端服务异常，设备配网失败
5.8 当天 15:00–20:00，App 后端服务出现异常，连锁拖累依赖后端的配网链路，设备配网失败。这是用户直接感知到的现象。
L2
直接原因
社区功能接口异常使用 Redis，导致 Redis 满载
App 后端服务的社区功能接口存在对 Redis 的异常使用方式，在当日流量下迅速把 Redis 打满。Redis 作为后端核心依赖一旦满载，包括设备配网在内的多个后端依赖功能均受到波及。
L3
根本原因
非核心业务可拖垮 IoT 核心功能，缺少服务降级保障
社区属于非核心业务，却能把配网、设备控制等 IoT 核心功能一同拖下水 —— 反映出当时缺少按业务优先级的服务降级与隔离机制。后端一旦出现压力，没有兜底策略优先保住 IoT 核心链路的可用性。
组织调整 · 团队架构整合

复盘后我们调整了组织架构，把原 IoT 云服务团队纳入 App 生态中心。后续 App 端到云端会作为一个团队整体对 App 稳定性负责，避免出现"端云分离导致的监控盲区"。

01
搭建接口稳定性监控平台
直击根因 · 解决"没有慢查询监控"
已上线

基于 Grafana + Prometheus 搭建接口稳定性监控平台。每个接口一旦达到阈值或 CPU 占满，监控会及时看到并触发告警。监控覆盖 JVM、Spring Boot 应用、数据库查询等关键维度。

告警通知示例：接口 500 报错（已恢复）
02
云端接口历史债清理
系统性排雷 · 不让"漏加索引"再重演
进行中

对所有云端接口进行历史债清理 —— 共 700+ 接口（其中涉及 IoT 的 349 个）、30 个定时任务，全部通过 AI 初步筛查 + 人工审核双层把关，把有问题的接口逐步排雷优化。

当前进度：AI SKILL 调教已完成；定时任务相关风险排查已完成，正在整改中；接口侧整改有序推进。
↗ API & Job 代码 · AI 扫描评审指导文档
03
按业务优先级启动逐级熔断降级机制
兜底保护 · 非核心服务不能拖垮核心
已上线

所有接口按业务优先级启动逐级熔断与降级机制。当非核心服务（如社区）出问题时，不会导致配网、设备控制等 IoT 核心服务连锁故障；若极端情况下核心服务仍受影响，将挂牌告知用户，避免无声失效。

该机制已在 5.8 事故后上线，后续如出现类似后端压力场景，系统会自动优先保障配网、设备控制等 IoT 核心功能的可用性。
